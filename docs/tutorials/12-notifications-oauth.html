1 Notifications Multi-Canal Intermediate

The notification system allows sending messages through multiple channels simultaneously: database, email (SMTP), and Slack (webhook). Each notification defines its channels and content for each.

Create a notification

A notification implements NotificationInterface and defines its channels via via():

namespace App\Notifications;

use Framework\Notification\NotificationInterface;
use Framework\Notification\MailMessage;
use Framework\Notification\SlackMessage;

class OrderShipped implements NotificationInterface
{
    public function __construct(
        private array $order
    ) {}

    // Channels used for this notification
    public function via(): array
    {
        return ['database', 'mail', 'slack'];
    }

    // Content stored in database
    public function toDatabase(): array
    {
        return [
            'title' => 'Order shipped',
            'body'  => "Your order #{$this->order['id']} has been shipped.",
            'url'   => "/orders/{$this->order['id']}",
        ];
    }

    // Email content (fluent API)
    public function toMail(): MailMessage
    {
        return (new MailMessage)
            ->subject('Order shipped')
            ->line("Your order #{$this->order['id']} is on its way.")
            ->action('Track delivery', "https://app.example.com/orders/{$this->order['id']}")
            ->line('Thank you for your trust!');
    }

    // Slack message (fluent API)
    public function toSlack(): SlackMessage
    {
        return (new SlackMessage)
            ->to('#orders')
            ->text("Order #{$this->order['id']} shipped to {$this->order['customer']}")
            ->emoji(':package:');
    }
}
Available channels
ChannelClassTransportConfiguration
database DatabaseChannel Table notifications None (table auto-created)
mail MailChannel SMTP MAIL_HOST, MAIL_PORT, MAIL_USERNAME, MAIL_PASSWORD
slack SlackChannel Webhook HTTP SLACK_WEBHOOK_URL
Send and read notifications

Use the HasNotifications trait on your User model:

use Framework\Notification\HasNotifications;

class User extends Model
{
    use HasNotifications;
}

// Send a notification
$user = User::find(1);
$user->notify(new OrderShipped($order));

// Read unread notifications
$unread = $user->unreadNotifications();

// Mark as read
$user->markNotificationAsRead($notificationId);

// All notifications (read + unread)
$all = $user->notifications();
Async sending. Email and Slack sending can be deferred by combining with the Job Queue: the notification is saved to DB immediately, then an async job handles SMTP and Slack sending.

2 Broadcasting SSE Advanced

Broadcasting uses Server-Sent Events (SSE) to push real-time data to browser clients. Events are published via Redis and broadcast by a dedicated SSE endpoint.

Server-side: publish an event 
use Framework\Broadcasting\Broadcaster;

// Publish an event to a channel
Broadcaster::broadcast('orders', 'shipped', [
    'order_id' => 42,
    'status'   => 'shipped',
    'eta'      => '2026-03-23',
]);

// Publish to multiple channels
Broadcaster::broadcast(['orders', 'admin'], 'order.updated', $data);
Attribut #[Broadcast] sur les evenements

Annotez vos classes d'evenements pour qu'ils soient automatically diffuses :

namespace App\Events;

use Framework\Broadcasting\Broadcast;

#[Broadcast('orders')]
class OrderShipped
{
    public function __construct(
        public readonly int $orderId,
        public readonly string $status = 'shipped',
    ) {}

    // Data sent to the SSE client
    public function broadcastWith(): array
    {
        return [
            'order_id' => $this->orderId,
            'status'   => $this->status,
        ];
    }
}
Publication automatique. Quand un evenement annote #[Broadcast] est dispatche via Event::dispatch(), le Broadcaster le publie automatically sur le canal Redis indique.
SSE Endpoint

The framework exposes a native SSE endpoint that listens to Redis channels and relays events to connected clients:

Client JS                          Serveur SSE                         Redis
    │                                  │                                 │
    │  GET /events/stream?channels=    │                                 │
    │      orders,notifications        │                                 │
    │─────────────────────────────────►│                                 │
    │                                  │  SUBSCRIBE orders               │
    │                                  │  SUBSCRIBE notifications        │
    │                                  │────────────────────────────────►│
    │                                  │                                 │
    │                                  │  heartbeat (every 15s)          │
    │◄─────────── :ping ──────────────│                                 │
    │                                  │                                 │
    │                                  │  PUBLISH orders {event_data}    │
    │                                  │◄────────────────────────────────│
    │◄── data: {"order_id":42,...} ───│                                 │
    │                                  │                                 │

// The SSE endpoint sends a heartbeat every 15 seconds
// to keep the connection alive:
// : ping\n\n

// SSE event format:
// event: shipped
// data: {"order_id":42,"status":"shipped","eta":"2026-03-23"}
// \n
Client-side: JavaScript 
// Connect to the SSE stream with desired channels
const source = new EventSource('/events/stream?channels=orders,notifications');

// Listen for a specific event type
source.addEventListener('shipped', (e) => {
    const data = JSON.parse(e.data);
    console.log(`Order #${data.order_id} shipped`);
});

// Ecouter tous les messages (evenement 'message' by default)
source.onmessage = (e) => {
    console.log('New event:', JSON.parse(e.data));
};

// Error handling and automatic reconnection
source.onerror = (e) => {
    console.warn('SSE connection lost, reconnecting automatically...');
};
Reconnexion automatique. Le navigateur reconnecte automatically le flux SSE en cas de coupure. Le serveur envoie un Last-Event-ID pour reprendre la ou le client s'est arrete, evitant la perte de messages.

3 OAuth / Socialite Intermediate

The OAuth module allows connecting users via external providers (Google, GitHub). The OAuth2 flow is entirely managed by the framework: redirect, code exchange, account creation, and JWT generation.

Complete OAuth2 flow

     │                    │                            │
     │  GET /auth/google/ │                            │
     │    redirect        │                            │
     │───────────────────►│                            │
     │                    │  302 Redirect              │
     │                    │  → accounts.google.com/    │
     │                    │    o/oauth2/auth?           │
     │                    │    client_id=...            │
     │◄───────────────────│                            │
     │                    │                            │
     │  Google Login  │                            │

     │                    │                            │
     │  GET /auth/google/ │                            │
     │    callback?code=  │                            │
     │───────────────────►│                            │
     │                    │  POST /oauth/token          │
     │                    │  (exchange code → token)     │

OAuth Routes 
// Redirect to the OAuth provider
// GET /auth/google/redirect
// → Redirects the user to the Google login page

// Callback after authentication
// GET /auth/google/callback?code=abc123&state=xyz
// → Exchanges the code, creates/finds the user, returns a JWT

// Same principle for GitHub
// GET /auth/github/redirect
// GET /auth/github/callback
OAuthManager Configuration 
use Framework\OAuth\OAuthManager;

// Get an OAuth driver
$google = OAuthManager::driver('google');

// Generate the redirect URL
$redirectUrl = $google->redirect();

// Exchange the code in the callback
$oauthUser = $google->callback($code);
// $oauthUser contains: id, email, name, avatar

// Create or find the local user
$user = User::firstOrCreate(
    ['email' => $oauthUser->email],
    [
        'name'     => $oauthUser->name,
        'avatar'   => $oauthUser->avatar,
        'provider' => 'google',
    ]
);

4 Deploy CLI Beginner

The deploy command automates the complete deployment cycle: Docker image build, push to registry, and Kubernetes rollout. All in a single command.

Standard deployment 
# Complete deployment (build + push + rollout)
./forge deploy

# Deployment with a specific tag
./forge deploy --tag=v2.3.1

# Dry-run mode: shows commands without executing them
./forge deploy --dry-run
./forge deploy
     │
     ├─► 1. Git tag detection (or --tag)
     │      git describe --tags --abbrev=0
     │
     ├─► 2. Docker image build
     │      docker build -t registry/image:tag .
     │
     ├─► 3. Push to the registry
     │      docker push registry/image:tag
     │
     └─► 4. Kubernetes rollout
            kubectl set image deployment/app
              app=registry/image:tag
              -n namespace
Options and flags
FlagDescriptionDefaut
--tag=v1.0.0 Docker image tag Dernier tag git detecte automatically
--dry-run Shows commands without executing them false
Environment variables 
# Docker Registry (e.g. Docker Hub, GCR, ECR)
DEPLOY_REGISTRY=registry.example.com

# Image name
DEPLOY_IMAGE=myapp/api

# Kubernetes namespace
DEPLOY_NAMESPACE=production
Automatic tag detection. If --tag is not specified, the command uses git describe --tags --abbrev=0 to detect the last git tag. If no tag exists, the short commit hash is used as a fallback.
Full example 
# 1. Create a git tag
git tag v2.4.0
git push --tags

# 2. Deployer (le tag est detecte automatically)
./forge deploy

# Output:
# [deploy] Tag detected: v2.4.0
# [deploy] Building registry.example.com/myapp/api:v2.4.0 ...
# [deploy] Pushing to registry.example.com ...
# [deploy] Rolling out to namespace 'production' ...
# [deploy] Done. Deployment successful.
Prerequisites. The docker and kubectl tools must be installed and configured on the machine running the command. Make sure the Kubernetes context points to the correct cluster.
