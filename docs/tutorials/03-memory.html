Memory & Monitoring

Detecting memory leaks, analyzing trends and monitoring your workers in production

Tutorials / 03 — Memory

1 Memory Trend Analysis Intermediate

WorkerStats records memory consumption on each request and calculates a trend automatically. This allows detecting leaks before they crash the worker.

How it works

On each request, beforeRequest() captures memory_get_usage(). After enough samples, the system compares the start and end of the window to determine the trend.

// In WorkerStats::beforeRequest()
$this->memorySnapshots[] = memory_get_usage(true);

// Keep only the last N samples
if (count($this->memorySnapshots) > $this->windowSize) {
    array_shift($this->memorySnapshots);
}
Trend values
TrendMeaningAction
stableMemory stays constant between requestsNone — all is well
growingMemory increases steadily (probable leak)Identify the source, see section 2
spikyOccasional spikes but returns to normalCheck heavy requests (uploads, reports)
insufficient_dataNot enough requests to conclude (< 10 samples)Wait for more traffic
curl examples 
# Get worker stats
curl http://localhost:8080/debug/worker | jq '.memory'

# Monitor the trend continuously (every 5s)
watch -n 5 'curl -s http://localhost:8080/debug/worker | jq .memory.trend'

# Alert if the trend is "growing"
TREND=$(curl -s http://localhost:8080/debug/worker | jq -r '.memory.trend')
if [ "$TREND" = "growing" ]; then
    echo "ALERT: memory leak detected!"
fi

2 Common Memory Leak Sources Advanced

In worker mode, the PHP process never dies. A forgotten allocation on each request will eventually saturate memory.

Unbounded static arrays

A static array that grows on each request is the #1 cause of leaks.

Guaranteed leak
class Cache
{
    private static array $store = [];

    public static function set($k, $v)
    {
        self::$store[$k] = $v;
        // Never shrinks!
        // Request 1: 10 entries
        // Request 10000: 100000 entries
    }
}
With LRU limit
class Cache
{
    private static array $store = [];
    private const MAX = 1000;

    public static function set($k, $v)
    {
        self::$store[$k] = $v;
        if (count(self::$store) > self::MAX) {
            array_shift(self::$store);
        }
    }
}
Closures capturing large data

A closure that captures a large variable by reference will prevent the garbage collector from freeing it.

Implicit capture
$bigData = file_get_contents('huge.csv'); // 50 MB

$processor = function() use ($bigData) {
    // $bigData stays in memory
    // even after the request!
    return str_word_count($bigData);
};

// If $processor is stored in a static...
Registry::add($processor); // 50 MB leak
Process and release
$bigData = file_get_contents('huge.csv');
$result = str_word_count($bigData);
unset($bigData); // Release immediately

// The closure only captures the result (int)
$processor = fn() => $result;
Event listeners registered per request

Registering a listener in a controller means it accumulates on each request.

In a controller (dangerous)
class UserController
{
    public function create($req)
    {
        // Adds 1 listener on EACH request!
        Event::listen('user.created',
            fn($e) => sendEmail($e)
        );
        // Request 500 = 500 emails sent
    }
}
At boot (correct)
// app/Listeners/SendWelcomeEmail.php
// Discovered automatically at boot
class SendWelcomeEmail
{
    public function handle(UserCreated $e)
    {
        sendEmail($e->user);
    }
}

// 1 single listener, regardless of the number
// of requests processed
Simple rule: if your code calls Event::listen() in a file under app/Controllers/, it's a leak. Move it to app/Listeners/.

3 MAX_REQUESTS Configuration Beginner

MAX_REQUESTS limits the number of requests a worker handles before restarting. It's a safety net against memory leaks.

Environment variable 
# .env or docker-compose.yml
MAX_REQUESTS=500    # Restart the worker after 500 requests

# In production with a stable app:
MAX_REQUESTS=5000   # Fewer restarts = better perf

# To disable (not recommended):
MAX_REQUESTS=0      # The worker will never restart
Recommended strategy
PhaseValueReason
Development50Detect leaks quickly
Staging / QA500Good safety/performance compromise
Production (start)500Safe starting value
Production (stable)5000Increase gradually if trend is stable
Strategy: Start at 500. Monitor the memory trend via /debug/worker. If it stays stable for several days, gradually increase to 5000. If it goes to growing, reduce and investigate.
Reading it in the code 
// App::runWorker()
$maxRequests = (int) ($_ENV['MAX_REQUESTS'] ?? 500);

for ($i = 0; $i < $maxRequests || $maxRequests === 0; ++$i) {
    // Handle requests...
}

// When the loop ends, FrankenPHP restarts the worker
Logger::info("Worker restart after {$i} requests");

Docker logs (stderr)

In a Docker environment, logs are also written to stderr to be captured by docker logs.

# View container logs
docker logs -f my-frankenphp-app

# Filter errors
docker logs my-frankenphp-app 2>&1 | grep "ERROR"

# Logs from the last 10 minutes
docker logs --since 10m my-frankenphp-app
Docker best practice: Don't mount a volume for var/logs/ in production. Use docker logs and a centralized collector (ELK, Loki, Datadog).

7 Database in Worker Mode Intermediate

Managing connections and transactions in worker mode is critical. An oversight can block all subsequent requests.

Why flush() is critical

Imagine this scenario without flush():

// Request 1: starts a transaction
$db->beginTransaction();
$db->execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1");
// Exception thrown here! No commit, no rollback.
throw new Exception('Payment gateway timeout');

// ─── Without flush() ───
// The transaction stays OPEN
// Row id=1 is LOCKED (row lock)

// Request 2: another user
$db->execute("SELECT * FROM accounts WHERE id = 1");
// BLOCKED! Waiting for request 1's lock...
// Timeout after 30 seconds → 500 error

// Requests 3, 4, 5...: same problem
// The entire worker is paralyzed
Catastrophic scenario: Without flush(), an unfinished transaction locks database rows. All subsequent requests touching those rows are blocked. The worker becomes unusable. 
// What flush() does:
public function flush(): void
{
    foreach ($this->connections as $conn) {
        if ($conn->inTransaction()) {
            $conn->rollBack();    // Rollback any pending transaction
        }
    }
    $this->connections = [];       // Close all connections
}
Automatic reconnection

MySQL/PostgreSQL connections have a timeout (wait_timeout). In worker mode, a connection kept too long is closed by the server.

// The DatabaseManager handles reconnection
public function getConnection(string $name): PDO
{
    if (isset($this->connections[$name])) {
        try {
            // Test if the connection is still alive
            $this->connections[$name]->query('SELECT 1');
        } catch (PDOException) {
            // Connection lost → create a new one
            unset($this->connections[$name]);
        }
    }

    return $this->connections[$name]
        ??= $this->createConnection($name);
}
Good news: Our DatabaseManager handles all of this automatically. The flush() in the worker cleanup guarantees a clean state between each request, and reconnection is transparent.
