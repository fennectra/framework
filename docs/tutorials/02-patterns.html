Patterns & Architecture

Writing worker-safe code: singletons, controllers, models and middleware

Tutorials / 02 — Patterns

1 Singleton vs Instance Intermediate

In a worker, singletons live forever. This is a strength (reused connections, config loaded once) but also a risk (leaking state). The rule: singleton for stateless services, fresh instance for anything carrying request-scoped state.

Framework services table
ServiceRegistrationPersists?Why
DatabaseManagersingleton()YESManages the PDO connection pool, cleanup via flush()
JwtServicesingleton()YESStateless: encode/decode with a fixed key
EventDispatchersingleton()YESListeners loaded at boot, never modified
WorkerStatsinstance()YESObject created before the loop, injected as-is
Loggersingleton()YESStateless: writes to a file/stream
Controllersbind() implicitNONew instance on each request via autowiring
ModelsNot registeredNOCreated on demand, never cached
singleton() vs bind() vs instance() 
// singleton() — created ONCE, reused forever
$container->singleton(JwtService::class, function ($c) {
    return new JwtService($c->get('config')['jwt_secret']);
});

// bind() — new instance on EVERY get()/make()
$container->bind(QueryBuilder::class, function ($c) {
    return new QueryBuilder($c->get(DatabaseManager::class));
});

// instance() — already constructed object, injected as-is
$stats = new WorkerStats();
$container->instance(WorkerStats::class, $stats);
get() vs make() — get() returns the singleton if it exists, otherwise resolves via bind or autowiring. make() forces a new instance on every call, even for a singleton.

2 Controllers — instance-scoped, safe by default Beginner

Controllers are never singletons. The Container creates a new instance on each request via autowiring. You can therefore store state in $this without risk of leaking.

Example: a typical controller 
namespace App\Controllers;

use Framework\Core\DatabaseManager;
use Framework\Services\JwtService;

class UserController
{
    // Injected by the Container — fresh refs on each request
    public function __construct(
        private DatabaseManager $db,
        private JwtService $jwt,
    ) {}

    public function show(int $id): void
    {
        $user = User::find($id);

        if (!$user) {
            throw new HttpException(404, 'User not found');
        }

        json($user->toArray());
    }
}
Worker-safe by default. The controller is instantiated by Container::get() on each request. Its properties do not survive between requests.
Trap: never cache a controller as a singleton
Dangerous
// NEVER! The controller survives between requests
$container->singleton(
    UserController::class,
    fn() => new UserController(...)
);

// Request 1: $this->data = user A
// Request 2: $this->data still contains user A!
Correct
// The Container autowires automatically
// No need to register controllers
$controller = $container->get(
    UserController::class
);
// New instance on each request

3 Models & QueryBuilder — no static cache Intermediate

Models extend Framework\Core\Model with a static $table. They are created on demand, never cached in the Container. The QueryBuilder is also a fresh instance on each use.

Anatomy of a Model 
namespace App\Models;

use Framework\Core\Model;

class User extends Model
{
    protected static string $table = 'users';
    protected static string $primaryKey = 'id';
    protected static bool $timestamps = true;
    protected static bool $softDeletes = false;

    protected static array $casts = [
        'id'    => 'int',
        'email' => 'string',
        'active' => 'bool',
    ];
}
Why static $table is safe? The Model's static properties ($table, $primaryKey, $casts) are immutable configuration, not request-scoped state. They are defined in the class and never change.
QueryBuilder — fresh instance, zero leaks 
// Each call creates a new QueryBuilder
$users = User::where('active', 1)
    ->orderBy('created_at', 'DESC')
    ->limit(10)
    ->get();

// The QueryBuilder is NOT cached
// No WHERE clause "leaking" to the next request

$user = User::find(42);           // New internal QB
$admins = Role::where('name', 'admin')->first(); // New QB
Anti-pattern: static result cache
class User extends Model
{
    private static array $cache = [];

    public static function findCached(int $id)
    {
        // The cache survives between requests!
        // Memory leak + stale data
        return self::$cache[$id]
            ??= self::find($id);
    }
}
Correct: always query the DB
// No static cache in Models
// Connections are pooled by DatabaseManager
$user = User::find($id);

// If you need caching, use
// a dedicated service registered as singleton
// with a clear invalidation strategy

4 Adding a service to the Container Intermediate

When you create a new service, the key question is: does this service carry request-scoped state?

Decision tree
Does my service carry request-scoped state? │ ├── NO (stateless or config-only) │ └── singleton() — one instance forever │ Examples: JwtService, Logger, EventDispatcher │ ├── YES, but the object already exists before the loop │ └── instance() — inject the object as-is │ Example: WorkerStats (created before the loop) │ └── YES, each request needs a fresh object └── bind() — new instance on every get() Examples: Controllers, FormValidator
Full example: adding a CacheService 
namespace App\Services;

class CacheService
{
    private array $store = [];

    public function get(string $key): mixed
    {
        return $this->store[$key] ?? null;
    }

    public function set(string $key, mixed $value, int $ttl = 60): void
    {
        $this->store[$key] = [
            'value' => $value,
            'expires' => time() + $ttl,
        ];
    }

    public function flush(): void
    {
        $this->store = [];
    }
}
// In your bootstrap / ServiceProvider:
$container->singleton(CacheService::class, function () {
    return new CacheService();
});

// The cache persists between requests (intentional!)
// But watch out for memory — implement a TTL or max-size
Memory warning. A singleton that accumulates state (like an in-memory cache) must have a cleanup strategy: TTL, max size, or periodic flush. Otherwise, memory grows indefinitely and the worker will be killed by the OS.
Registration with autowiring 
// If your service only has typed dependencies,
// the Container resolves it automatically:
$mailer = $container->get(MailService::class);
// The Container reads the constructor via Reflection,
// resolves Logger, DatabaseManager, etc. automatically

// To force singleton mode with autowiring:
$container->singleton(MailService::class, MailService::class);
// Passing a string: the Container autowires it and caches it

5 Middleware — stateless pattern Advanced

Middleware are singletons loaded at boot via addGlobalMiddleware(). They live forever. They must NEVER store request-scoped state in their properties.

Correct middleware: purely functional 
namespace App\Middleware;

use Framework\Services\JwtService;

class AuthMiddleware
{
    public function __construct(
        private JwtService $jwt,
    ) {}

    public function handle(): bool
    {
        // Read from superglobals (reset by FrankenPHP)
        $header = $_SERVER['HTTP_AUTHORIZATION'] ?? '';

        if (!str_starts_with($header, 'Bearer ')) {
            throw new HttpException(401, 'Missing token');
        }

        $token = substr($header, 7);
        $payload = $this->jwt->decode($token);

        // Store in $_REQUEST (cleaned by the worker)
        $_REQUEST['__auth_user'] = $payload;

        return true;  // Continue the chain
    }
}
Anti-pattern: state in middleware
Dangerous: request-scoped property
class AuthMiddleware
{
    // DANGER: survives between requests!
    private ?array $currentUser = null;

    public function handle(): bool
    {
        $this->currentUser = $payload;
        return true;
    }

    public function getUser(): ?array
    {
        // Request 2 without token: returns
        // the user from request 1!
        return $this->currentUser;
    }
}
Correct: pass via $_REQUEST
class AuthMiddleware
{
    // No request-scoped property

    public function handle(): bool
    {
        $_REQUEST['__auth_user'] = $payload;
        return true;
    }
}

// In the controller:
$user = $_REQUEST['__auth_user'] ?? null;
// Cleaned by the worker in the finally block
Absolute rule. A middleware must never write request-scoped state to $this. Use $_REQUEST (cleaned in the worker's finally block) or an attribute on the Request object.
