1 JWT Authentication Beginner

The framework uses JSON Web Tokens (JWT) to authenticate API requests. The flow is simple: obtain a token, then send it with each request.

Obtain a token

Send a POST /token with your credentials to receive a signed JWT:

# Request
POST /token
Content-Type: application/json

{
    "email": "admin@example.com",
    "password": "secret"
}

# Response
{
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in": 3600
}
Use the token

Add the token in the Authorization header of each protected request:

GET /api/users
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

The Auth middleware validates the JWT signature, verifies that the token exists in the database (protection against revocation), then stores the user for the controller.

Retrieve the current user

In any protected controller, use Auth::user():

use App\Middleware\Auth;

class ProfileController
{
    public function show(): array
    {
        $user = Auth::user();          // array with id, email, role...

        return [
            'id'    => $user['id'],
            'email' => $user['email'],
            'role'  => $user['role'],
        ];
    }
}
Worker-safe. The user is stored in $_REQUEST['__auth_user'], automatically cleaned by the worker between each request. No risk of leaking.

2 RBAC — Role-Based Access Intermediate

The Auth middleware accepts one or more authorized roles. If the user does not have the correct role, an HttpException 403 is thrown.

Protect a route with a role 
use App\Middleware\Auth;
use App\Controllers\AdminController;

// Single authorized role
$router->get('/admin', [AdminController::class, 'index'], [[Auth::class, ['admin']]]);

// Multiple authorized roles
$router->get('/dashboard', [DashboardController::class, 'index'], [[Auth::class, ['admin', 'manager']]]);
Protect a route group

Use group() to apply the Auth middleware to multiple routes at once:

$router->group(['middleware' => [[Auth::class, ['admin', 'manager']]]], fn($r) =>
    $r->get('/users',    [UserController::class, 'index'])
      ->post('/users',   [UserController::class, 'store'])
      ->delete('/users/{id}', [UserController::class, 'destroy'])
);
Internal workings. The middleware calls Auth::handle($allowedRoles). It validates the JWT, loads the user from the DB, then compares $user['role'] with the array of authorized roles via in_array().

3 Container DI Intermediate

The Container is the heart of the architecture. It manages the creation and injection of services throughout the application.

The 5 essential methods
MethodDescriptionUsage
bind($abstract, $concrete)Registers a factory — new instance on each callStateless services
singleton($abstract, $concrete)Registers a singleton — instantiated onceConnections, configs
instance($abstract, $object)Registers an already constructed objectBoot-time objects
get($abstract)Resolves a dependency (singleton-aware)Retrieve a service
make($abstract, $params)Always creates a new instance (ignores the cache)Unique instances
Register a service 
use Framework\Core\Container;
use Framework\Core\Env;

$container = Container::getInstance();

// Singleton: created once, reused everywhere
$container->singleton(StripeService::class, fn() =>
    new StripeService(Env::get('STRIPE_KEY'))
);

// Bind: new instance on each resolution
$container->bind(PdfGenerator::class, fn() =>
    new PdfGenerator('/tmp/pdf')
);

// Instance: already constructed object
$container->instance('config', ['debug' => true]);
Autowiring — automatic injection

The Container automatically resolves constructor dependencies via type-hint. No manual configuration needed:

class OrderController
{
    public function __construct(
        private StripeService $stripe,  // Automatically injected
        private PdfGenerator $pdf,     // Automatically injected
    ) {}

    public function invoice(int $id): array
    {
        $charge = $this->stripe->getCharge($id);
        $this->pdf->generate($charge);
        return ['status' => 'generated'];
    }
}
Recursive resolution. If StripeService itself has dependencies in its constructor, the Container resolves them too, recursively. Default values and nullable types are supported.

4 Event Dispatcher Intermediate

The event system allows decoupling components. A service dispatches an event, listeners react without direct coupling.

Dispatch an event 
use Framework\Core\Event;

// With a string name + payload
Event::dispatch('user.created', $user);

// With an event object (recommended)
Event::dispatch(new UserCreated($user));
Listen for an event 
// Inline listener (at boot only!)
Event::listen('user.created', function ($user) {
    error_log("New user: " . $user['email']);
});

// One-shot listener (executed once)
Event::once('app.ready', fn() => error_log('Application started'));
Worker rule: never Event::listen() in a controller. Listeners accumulate on each request. Request 1: 1 listener. Request 100: 100 identical listeners. Place them at boot or in app/Listeners/.
Attribute #[Listener] — automatic discovery

Create a class in app/Listeners/ with the #[Listener] attribute. The framework discovers it automatically at boot:

namespace App\Listeners;

use Framework\Attributes\Listener;
use App\Events\UserCreated;

// Automatic discovery via discoverListeners()
#[Listener(UserCreated::class, priority: 0)]
class SendWelcomeEmail
{
    public function handle(UserCreated $event): void
    {
        // Send the welcome email
        mail($event->user['email'], 'Welcome!', '...');
    }
}

Discovery is triggered at application boot:

$dispatcher = Container::getInstance()->get(EventDispatcher::class);
$dispatcher->discoverListeners(__DIR__ . '/app/Listeners');

5 Event Brokers Advanced

The dispatcher supports 3 brokers: sync (default), redis, and database. Local listeners are always executed immediately; the redis and database brokers add an external publication for async processing.

Broker comparison
BrokerExecutionDependencyPersistenceUse case
sync Immediate, same process None Non Dev, simple logic
redis Immediate + Redis publication Extension php-redis Volatile (Redis) Microservices, real-time
database Immediate + DB insertion Table events Durable (SQL) Audit, reliable queue
.env Configuration 
# Broker selection
EVENT_BROKER=sync          # sync | redis | database

# Redis configuration (if EVENT_BROKER=redis)
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=null
REDIS_DB=0
REDIS_PREFIX=app:events:

# Database configuration (if EVENT_BROKER=database)
EVENT_DB_CONNECTION=default
EVENT_DB_TABLE=events
Required SQL table for database broker 
CREATE TABLE events (
    id         SERIAL PRIMARY KEY,
    event      VARCHAR(255) NOT NULL,
    payload    JSONB,
    status     VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT NOW(),
    processed_at TIMESTAMP NULL
);
Instantiation with driver 
use Framework\Core\EventDispatcher;

// Automatic creation based on EVENT_BROKER
$dispatcher = EventDispatcher::withDriver(Env::get('EVENT_BROKER', 'sync'));

// Check the active driver
echo $dispatcher->driverName();  // "sync", "redis", ou "database"

Exercise 1 : Route protegee par role Beginner

Goal: Create a route GET /articles/create accessible only to users with the editor role.

Hints:

  • Use the Auth middleware with the 'editor' role
  • The controller should return ['form' => 'article_create']
  • Test with a token from a user with a different role — you should get a 403
Solution 
// routes/api.php
use App\Middleware\Auth;
use App\Controllers\ArticleController;

$router->get(
    '/articles/create',
    [ArticleController::class, 'create'],
    [[Auth::class, ['editor']]]
);

// app/Controllers/ArticleController.php
class ArticleController
{
    public function create(): array
    {
        return ['form' => 'article_create'];
    }
}
Exercise 2 : Listener de connexion Intermediate

Goal: Create a listener that logs each user login. When the user.login event is dispatched, write the email and date to a log file.

Hints:

  • Create a class in app/Listeners/ with the #[Listener] attribute
  • Use file_put_contents() with FILE_APPEND
  • Dispatch the event in the TokenController after successful authentication
Solution 
// app/Listeners/LogUserLogin.php
namespace App\Listeners;

use Framework\Attributes\Listener;

#[Listener('user.login')]
class LogUserLogin
{
    public function handle(mixed $payload): void
    {
        $line = sprintf(
            "[%s] Login: %s\n",
            date('Y-m-d H:i:s'),
            $payload['email'] ?? 'unknown'
        );
        file_put_contents(__DIR__ . '/../../storage/logs/logins.log', $line, FILE_APPEND);
    }
}

// In TokenController::generate(), after token creation:
Event::dispatch('user.login', ['email' => $user['email'], 'ip' => $_SERVER['REMOTE_ADDR']]);
Exercise 3 : Service custom dans le Container Advanced

Goal: Create a NotificationService service, register it as a singleton in the Container, and inject it automatically in a controller via autowiring.

Hints:

  • The service should have a constructor that receives an API key from Env::get()
  • Register it with singleton() at boot
  • The controller should declare the type-hint in its constructor — the Container does the rest
Solution 
// app/Services/NotificationService.php
namespace App\Services;

class NotificationService
{
    public function __construct(
        private string $apiKey
    ) {}

    public function send(string $to, string $message): bool
    {
        // Sending logic...
        return true;
    }
}

// At boot (index.php or bootstrap):
$container->singleton(NotificationService::class, fn() =>
    new NotificationService(Env::get('NOTIFICATION_API_KEY'))
);

// app/Controllers/AlertController.php
class AlertController
{
    public function __construct(
        private NotificationService $notifier  // Autowired!
    ) {}

    public function send(): array
    {
        $user = Auth::user();
        $this->notifier->send($user['email'], 'Test alert');
        return ['status' => 'sent'];
    }
}
