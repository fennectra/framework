1 Scheduler + Redis Lock Intermediate

The Scheduler allows defining scheduled tasks with a fluent API. Combined with a Redis lock, it guarantees that only one instance executes even in a Kubernetes cluster.

Configuration in app/Schedule.php

Define your scheduled tasks in the app/Schedule.php file using the fluent API:

namespace App;

use Framework\Scheduler\Schedule;

return function (Schedule $schedule): void {

    // Clean expired tokens every 5 minutes
    $schedule->call(fn() => Token::deleteExpired())
        ->everyFiveMinutes()
        ->name('tokens:cleanup');

    // Daily report at 2 AM
    $schedule->call(fn() => ReportService::daily())
        ->dailyAt('02:00')
        ->name('reports:daily');

    // External sync every hour
    $schedule->call(fn() => ExternalSync::run())
        ->hourly()
        ->name('sync:external');
};
Redis Lock — preventing duplicates in K8s

In a multi-pod environment (Kubernetes), multiple workers execute Scheduler::tick() simultaneously. The Scheduler uses a distributed Redis lock to guarantee that only one instance executes each task:

Pod A  ──► Scheduler::tick()  ──► Redis SET task:tokens:cleanup NX EX 300
                                        ✓ Lock acquired → execute the task

Pod B  ──► Scheduler::tick()  ──► Redis SET task:tokens:cleanup NX EX 300
                                        ✗ Lock exists → skip

Pod C  ──► Scheduler::tick()  ──► Redis SET task:tokens:cleanup NX EX 300
                                        ✗ Lock exists → skip

// The Redis lock uses SET NX EX (atomic)
// NX = only if the key does not exist
// EX = automatic expiration (TTL = task duration + margin)
$redis->set(
    "scheduler:lock:{$task->name()}",
    gethostname(),
    ['NX', 'EX' => 300]
);
Security./strong> Le TTL du lock est toujours superieur a la duree prevue de la tache. Si un pod crash, le lock expire automatically et un autre pod prend le relais.
Execution in the FrankenPHP worker

Scheduler::tick() est appele dans la boucle du worker FrankenPHP. Il est automatically throttle a 60 secondes pour ne pas surcharger le systeme :

// Dans le worker loop FrankenPHP
while (frankenphp_handle_request($handler)) {
    // HTTP request processing...

    // The tick checks if 60s have elapsed since the last check
    Scheduler::tick();  // Internal throttle: max 1x / 60s
}
Required environment variable. The scheduler only activates if SCHEDULER_ENABLED=1 is defined in your .env. Without this variable, tick() is a no-op.
Manual execution (one-shot)

To execute all scheduled tasks immediately (useful in dev or in a classic cron):

# Execute all eligible tasks now
./forge schedule:run

# List configured tasks
./forge schedule:list

2 Job Queue Intermediate

The job queue allows executing heavy tasks in the background (sending emails, image processing, external sync) without blocking the HTTP request.

Dispatch a job

Use Job::dispatch() to add a job to the queue:

use Framework\Queue\Job;

// Dispatch a job avec des parametres
Job::dispatch(SendEmail::class, [
    'user_id' => 1,
    'template' => 'welcome',
]);

// Dispatch with a delay (in seconds)
Job::dispatch(GenerateReport::class, [
    'month' => '2026-03',
])->delay(60);

// Dispatch to a specific queue
Job::dispatch(ProcessImage::class, [
    'path' => '/uploads/photo.jpg',
])->onQueue('images');
Consume jobs 
# Run the queue worker (daemon mode)
./forge queue:work

# Worker on a specific queue
./forge queue:work --queue=images

# Process a single job then exit
./forge queue:work --once
Drivers : Redis vs Database
DriverCommandAdvantagesDisadvantages
redis BLPOP queue:default 30 Ultra-fast, instant notification Possible loss if Redis crashes without persistence
database SELECT ... FOR UPDATE SKIP LOCKED Durable, no external dependency Slower, polling required 
# .env
QUEUE_DRIVER=redis     # redis | database
QUEUE_PREFIX=app:queue:
Driver Redis :
  Controller ──► LPUSH queue:default {job_payload}
                      │
  Worker     ──► BLPOP queue:default 30  ──► deserialize ──► execute

Driver Database :
  Controller ──► INSERT INTO jobs (queue, payload, available_at, ...)
                      │
  Worker     ──► SELECT * FROM jobs WHERE queue = 'default'
                   AND available_at <= NOW()
                   ORDER BY id
                   FOR UPDATE SKIP LOCKED
                   LIMIT 1
Retry and failure handling

Each job can define its retry policy:

namespace App\Jobs;

use Framework\Queue\AbstractJob;

class SendEmail extends AbstractJob
{
    public int $maxRetries = 3;       // Maximum number of attempts
    public int $retryDelay = 30;      // Delay between each retry (seconds)

    public function handle(): void
    {
        $user = User::find($this->data['user_id']);
        Mailer::send($user->email, $this->data['template']);
    }

    public function failed(\Throwable $e): void
    {
        // Called after maxRetries failures
        error_log("Job SendEmail failed: " . $e->getMessage());
    }
}

Jobs that fail after all attempts are moved to the failed_jobs table:

# View failed jobs
./forge queue:failed

# Retry a failed job by its ID
./forge queue:retry --id=5

# Retry all failed jobs
./forge queue:retry --all
Generate a job with the CLI 
# Create a new job in app/Jobs/
./forge make:job SendWelcomeEmail

This generates the app/Jobs/SendWelcomeEmail.php file with the base structure ready to use.

3 Feature Flags Beginner

Feature Flags allow enabling or disabling features without deployment. Ideal for progressive releases, A/B testing, and emergency killswitches.

Simple check 
use Framework\Feature\FeatureFlag;

// Check if a feature is active
if (FeatureFlag::enabled('new-api')) {
    // New API v2 logic
    return $this->handleV2($request);
}

// Fallback to old logic
return $this->handleV1($request);
Conditional flags (by role)

Flags can be targeted by user role for progressive deployment:

// Active only for admins
if (FeatureFlag::for('beta')->whenRole('admin')->enabled()) {
    // Beta interface reserved for admins
}

// Active for multiple roles
if (FeatureFlag::for('dashboard-v2')->whenRole('admin', 'manager')->enabled()) {
    // New dashboard
}
Enable / disable programmatically 
// Enable a feature
FeatureFlag::activate('dark-mode');

// Disable a feature
FeatureFlag::deactivate('dark-mode');
Redis cache (60s) + DB fallback. Flags are cached in Redis for 60 seconds. If Redis is unavailable, the system queries the database directly. Changes via activate() / deactivate() invalidate the cache immediately.
Management via CLI 
# Enable a flag
./forge feature enable dark-mode

# Disable a flag
./forge feature disable dark-mode

# List all flags and their status
./forge feature list

Example output of feature list :

4 State Machine Intermediate

The state machine allows modeling complex workflows (orders, tickets, approval processes) with validated transitions and automatic events.

Define a state machine

Use the #[StateMachine] attribute on your model and the HasStateMachine trait:

namespace App\Models;

use Framework\ORM\Model;
use Framework\StateMachine\HasStateMachine;
use Framework\StateMachine\StateMachine;

#[StateMachine(
    column: 'status',
    states: ['draft', 'pending', 'confirmed', 'shipped', 'delivered', 'cancelled'],
    transitions: [
        'submit'  => ['from' => 'draft',     'to' => 'pending'],
        'confirm' => ['from' => 'pending',   'to' => 'confirmed'],
        'ship'    => ['from' => 'confirmed', 'to' => 'shipped'],
        'deliver' => ['from' => 'shipped',   'to' => 'delivered'],
        'cancel'  => ['from' => ['draft', 'pending'], 'to' => 'cancelled'],
    ]
)]
class Order extends Model
{
    use HasStateMachine;

    protected string $table = 'orders';
}
Use transitions 
$order = Order::find(42);

// Check if a transition is possible
$order->canTransitionTo('shipped');    // true si etat = confirmed

// Perform the transition
$order->transitionTo('shipped');      // Changes the status + saves to DB

// List available transitions from the current state
$order->availableTransitions();
// Ex: si etat = 'pending' → ['confirm', 'cancel']
Invalid transition. Calling transitionTo() to an unauthorized state throws an InvalidTransitionException. Always use canTransitionTo() first in a user context.
Automatic events

Chaque transition dispatch automatically des evenements que vous pouvez ecouter :

use Framework\Core\Event;

// Event before the transition
Event::listen('order.transitioning:shipped', function ($order) {
    // Validate stock, prepare shipment...
});

// Event after the transition
Event::listen('order.transitioned:shipped', function ($order) {
    // Send a notification to the client
    Job::dispatch(SendShippingNotification::class, [
        'order_id' => $order->id,
    ]);
});
