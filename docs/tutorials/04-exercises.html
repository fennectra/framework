Exercise 1 — Find the worker bug Beginner

This StatsController works perfectly in PHP-FPM. In worker mode, users report that the call counter displays strange numbers. Find the bug.

class StatsController
{
    private static int $callCount = 0;

    public function index(): array
    {
        self::$callCount++;

        return [
            'status'  => 'ok',
            'calls'   => self::$callCount,
            'message' => 'Number of calls to this route',
        ];
    }
}

Question: Why does the counter increase indefinitely? What does the user see on the 500th request?

See the solution
The problem static $callCount persists between requests in worker mode. Each request handled by this thread increments the same counter. On the 500th request, the user sees "calls": 500 instead of "calls": 1.

In PHP-FPM, each request starts a new process: $callCount is always reset to 0. In worker mode, the class stays in memory.

Solutions:

  • If the counter is intentional (internal metrics): this is the correct behavior, but document it. The counter is per-thread, not global.
  • If it's a business counter: store it in the database or via Redis.
Dangerous
private static int $callCount = 0;
// Accumulates between requests!
Correct (business counter)
public function index(Database $db): array
{
    $db->exec(
        'UPDATE stats SET calls = calls + 1'
    );
    $count = $db->fetchOne(
        'SELECT calls FROM stats'
    );
    return ['calls' => $count];
}
Exercise 2 — Memory leak Intermediate

This audit service records each user action. After a few hours in production, the worker consumes 2 GB of RAM and ends up being killed by the OOM killer.

class AuditService
{
    private static array $logs = [];

    public static function log(string $action, array $context = []): void
    {
        self::$logs[] = [
            'time'    => microtime(true),
            'action'  => $action,
            'context' => $context,
            'memory'  => memory_get_usage(),
        ];
    }

    public static function getAll(): array
    {
        return self::$logs;
    }
}

Question: Why does memory explode? Propose 3 correction strategies.

See the solution
The problem static $logs is an array that grows on each request without ever being emptied. After 100,000 requests with rich contexts, the array occupies hundreds of MB.

3 correction strategies:

1. Write to database (recommended)

public static function log(string $action, array $context = []): void
{
    Database::table('audit_logs')->insert([
        'action'  => $action,
        'context' => json_encode($context),
        'time'    => microtime(true),
    ]);
}

2. Use a file Logger

public static function log(string $action, array $context = []): void
{
    $line = json_encode(['action' => $action, 'context' => $context]);
    file_put_contents('audit.log', $line . "\n", FILE_APPEND);
}

3. Limit in-memory buffer size

private static int $maxLogs = 100;

public static function log(string $action, array $context = []): void
{
    self::$logs[] = ['action' => $action, 'context' => $context];

    if (count(self::$logs) > self::$maxLogs) {
        self::$logs = array_slice(self::$logs, -self::$maxLogs);
    }
}
Rule In worker mode, never store unlimited data in a static property. If you need a buffer, set a maximum size and purge regularly.
Exercise 3 — Security flaw: privilege escalation Advanced

This middleware protects admin routes. A tester discovers that after an admin logs in, all subsequent users have access to admin routes. This is a critical vulnerability.

class AdminMiddleware
{
    private static bool $isAdmin = false;

    public function handle(array $request, callable $next): mixed
    {
        if (isset($request['user']) && $request['user']['role'] === 'admin') {
            self::$isAdmin = true;
        }

        if (!self::$isAdmin) {
            throw new HttpException(403, 'Forbidden');
        }

        return $next($request);
    }
}

Question: Explain the complete attack scenario. How to fix it?

See the solution
Attack scenario
  1. Request 1: Admin logs in. $isAdmin becomes true.
  2. Request 2: Regular user arrives. $isAdmin is still true (static persists).
  3. The if never sets $isAdmin back to false.
  4. Result: everyone is admin on this thread.

Fix: never cache an authorization state in a static property. Check on every request.

Critical vulnerability
private static bool $isAdmin = false;
// Never reset!
Correct
public function handle(array $request, callable $next): mixed
{
    // Check on every request, no caching
    $isAdmin = isset($request['user'])
        && $request['user']['role'] === 'admin';

    if (!$isAdmin) {
        throw new HttpException(403, 'Forbidden');
    }

    return $next($request);
}
Security rule Never use static to store authorization decisions. Each request must be evaluated independently. This is the most important rule in worker mode.
Exercise 4 — Orphaned transaction Intermediate

This controller creates a user with a profile. Sometimes, subsequent requests fail with "SQLSTATE: there is already an active transaction". Find out why.

class UserController
{
    public function create(Database $db, array $data): array
    {
        $db->exec('BEGIN');

        $userId = $db->insert('users', [
            'name'  => $data['name'],
            'email' => $data['email'],
        ]);

        // If $data['bio'] is missing, an exception is thrown here
        $db->insert('profiles', [
            'user_id' => $userId,
            'bio'     => $data['bio'],
        ]);

        $db->exec('COMMIT');

        return ['id' => $userId];
    }
}

Question: What happens if the profile insertion fails? Why is the next worker request affected?

See the solution
The problem If insert('profiles', ...) throws an exception, the COMMIT is never executed. The transaction stays open. The PDO connection persists in worker mode, so the next request using this connection inherits an open transaction.

Scenario:

  1. Request 1: BEGIN → INSERT users → exception → no COMMIT or ROLLBACK.
  2. The transaction stays open on the PDO connection.
  3. Request 2: attempts a BEGIN → "active transaction" error.
  4. Worse: if no BEGIN, writes occur in request 1's orphaned transaction.

Solution 1: try/catch with ROLLBACK

public function create(Database $db, array $data): array
{
    $db->exec('BEGIN');

    try {
        $userId = $db->insert('users', [...]);
        $db->insert('profiles', [...]);
        $db->exec('COMMIT');
    } catch (\Throwable $e) {
        $db->exec('ROLLBACK');
        throw $e;
    }

    return ['id' => $userId];
}

Solution 2: transaction() method (recommended)

public function create(Database $db, array $data): array
{
    return $db->transaction(function () use ($db, $data) {
        $userId = $db->insert('users', [...]);
        $db->insert('profiles', [...]);
        return ['id' => $userId];
    });
}
Our framework handles it The worker cleanup calls $dbManager->flush() which performs an automatic ROLLBACK on any active transaction. But it's better to handle it properly in the business code.
Exercise 5 — Design a worker-safe RateLimiter Advanced

Design an in-memory RateLimiter that works correctly in worker mode with these constraints:

  • In-memory storage (no Redis/DB) — acceptable for lightweight rate-limiting
  • Cleanup of expired entries to avoid memory leaks
  • Awareness that each thread has its own RateLimiter (no cross-thread sharing)
  • Configurable limit per IP: X requests per Y second window

Question: Implement the complete class with hit(), isAllowed(), and cleanup() methods.

See the complete solution 
class WorkerRateLimiter
{
    /** @var array<string, array{count: int, window_start: float}> */
    private array $hits = [];
    private int $lastCleanup = 0;

    public function __construct(
        private readonly int $maxRequests = 60,
        private readonly int $windowSeconds = 60,
        private readonly int $cleanupEvery = 100,  // Clean every N requests
        private readonly int $maxEntries = 10000,  // Absolute entry limit
    ) {}

    /**
     * Record a hit and return true if the request is allowed.
     */
    public function hit(string $ip): bool
    {
        $now = time();
        $this->maybeCleanup($now);

        if (!isset($this->hits[$ip])) {
            $this->hits[$ip] = [
                'count'        => 0,
                'window_start' => $now,
            ];
        }

        $entry = &$this->hits[$ip];

        // Window expired? Reset
        if (($now - $entry['window_start']) >= $this->windowSeconds) {
            $entry['count'] = 0;
            $entry['window_start'] = $now;
        }

        $entry['count']++;

        return $entry['count'] <= $this->maxRequests;
    }

    /**
     * Check without incrementing.
     */
    public function isAllowed(string $ip): bool
    {
        if (!isset($this->hits[$ip])) {
            return true;
        }

        $entry = $this->hits[$ip];
        $now = time();

        if (($now - $entry['window_start']) >= $this->windowSeconds) {
            return true;  // Window expired
        }

        return $entry['count'] < $this->maxRequests;
    }

    /**
     * Remove expired entries to prevent memory leaks.
     */
    public function cleanup(int $now): void
    {
        foreach ($this->hits as $ip => $entry) {
            if (($now - $entry['window_start']) >= $this->windowSeconds) {
                unset($this->hits[$ip]);
            }
        }

        // Ultimate protection against memory leaks
        if (count($this->hits) > $this->maxEntries) {
            $this->hits = array_slice($this->hits, -$this->maxEntries, preserve_keys: true);
        }

        $this->lastCleanup = $now;
    }

    private function maybeCleanup(int $now): void
    {
        if (($now - $this->lastCleanup) >= $this->cleanupEvery) {
            $this->cleanup($now);
        }
    }

    public function getStats(): array
    {
        return [
            'tracked_ips' => count($this->hits),
            'memory'      => memory_get_usage(),
        ];
    }
}
Limitations of in-memory rate-limiting Each worker thread has its own instance. With 4 threads, an attacker can make 4 x maxRequests requests per window. For strict rate-limiting, use Redis or a shared database.

Usage in the worker:

// Created at boot, before the loop
$limiter = new WorkerRateLimiter(maxRequests: 60, windowSeconds: 60);

// In the handler
$ip = $_SERVER['REMOTE_ADDR'];
if (!$limiter->hit($ip)) {
    http_response_code(429);
    echo json_encode(['error' => 'Too many requests']);
    return;
}

2 Worker-Ready Checklist

Before deploying in worker mode, verify each item:

  • No exit() or die() in application code
  • No static property containing request data (user, session, token)
  • No Event::listen() called in a controller or per-request middleware
  • All DB transactions are wrapped with try/catch or DB::transaction()
  • The worker cleanup removes manually added data from $_REQUEST
  • PDO connections are closed or rolled back via flush() after each request
  • Static arrays used as cache have a defined maximum size
  • The garbage collector is called periodically (gc_collect_cycles())
  • Configuration files are loaded at boot, not on each request
  • HTTP headers are set before any echo in the handler
  • Container singletons are stateless or properly reinitialized
  • Tests pass in worker mode (not just in PHP-FPM)
