1 Debug Profiler Intermediate

The Profiler is a diagnostic tool built into the framework. It collects detailed metrics for each HTTP request: SQL queries, dispatched events, middleware timing, memory consumption and N+1 problem detection.

Activation

The profiler is activated in two ways in your .env file:

# Method 1: explicit activation
PROFILER_ENABLED=1

# Method 2: automatic in development mode
APP_ENV=dev
Production. Ne jamais activer le profiler in production. Il ajoute un overhead sur chaque requete et expose des informations sensibles (requetes SQL, structure interne).
Viewing endpoints

Once activated, two endpoints are available:

EndpointDescription
GET /debug/profilerList of the last 50 profiled requests
GET /debug/profiler/{id}Complete detail of a specific request
JSON output structure

Each profile returns a JSON object with the following sections:

{
    "id":          "req_64a1b2c3",
    "method":      "GET",
    "uri":         "/api/users",
    "status":      200,
    "duration_ms": 12.4,
    "memory_mb":   2.1,
    "sql": {
        "count":   3,
        "time_ms": 4.2,
        "queries": [
            { "sql": "SELECT * FROM users", "time_ms": 1.8, "params": [] }
        ]
    },
    "events":      ["user.loaded", "response.sent"],
    "middlewares": [
        { "name": "Auth", "time_ms": 0.8 },
        { "name": "RateLimit", "time_ms": 0.3 }
    ],
    "warnings":    ["N+1 detected: SELECT * FROM posts WHERE user_id = ? (x5)"]
}
Collection hooks

Le profiler s'insere automatically dans les composants suivants pour collecter les metriques :

ComponentData collected
Database::query()SQL query, parameters, execution time
EventDispatcherEvent name, number of listeners, total time
MiddlewarePipelineMiddleware name, processing duration
ContainerResolved services, resolution time
N+1 Detection

The profiler analyzes SQL queries collected during a request. If the same SQL pattern (same structure, different parameters) is executed more than 3 times, an N+1 warning is issued.

// Example of N+1 problem:
foreach ($users as $user) {
    // Each iteration generates a separate SQL query!
    $posts = Post::query()->where('user_id', $user->id)->get();
}

// The profiler detects:
// "N+1 detected: SELECT * FROM posts WHERE user_id = ? (x12)"

// Solution: eager loading
$users = User::query()->with('posts')->get();
Seuil configurable. Le seuil by default est de 3 repetitions. Ce seuil est volontairement bas pour detecter les problemes le plus tot possible pendant le developpement.
InMemory Storage (ring buffer)

Profiles are stored in memory via InMemoryStorage, a circular buffer that keeps the last 50 requests. In worker mode (FrankenPHP), this buffer persists between requests of the same worker, providing history without disk I/O.

  Requete 1    Requete 2    ...    Requete 50    Requete 51
  ┌────────┐   ┌────────┐         ┌────────┐    ┌────────┐
  │ Profil │   │ Profil │   ...   │ Profil │    │ Profil │
  └───┬────┘   └───┬────┘         └───┬────┘    └───┬────┘
      │            │                   │             │
      v            v                   v             v
  ┌──────────────────────────────────────────────────────┐
  │          InMemoryStorage  (ring buffer = 50)         │
  │  [slot 0] [slot 1] ... [slot 49]                     │
  │     ^                                                │
  │     └── Request 51 overwrites Request 1                  │
  └──────────────────────────────────────────────────────┘
Integration in the pipeline

The ProfilerMiddleware must be placed first in the pipeline to measure the entire request cycle:

use Framework\Middleware\ProfilerMiddleware;
use Framework\Middleware\SecurityHeadersMiddleware;
use Framework\Middleware\RateLimitMiddleware;
use Framework\Middleware\RequestLoggingMiddleware;

// config/middleware.php
return [
    // Profiler first to capture the entire cycle
    ProfilerMiddleware::class,

    // Global middleware
    SecurityHeadersMiddleware::class,
    RequestLoggingMiddleware::class,
    RateLimitMiddleware::class,
];
Conditionnel. Le ProfilerMiddleware verifie automatically si le profiler est active. S'il ne l'est pas, il passe directement au middleware suivant sans aucun overhead.

2 Rate Limiting Beginner

Le Rate Limiting protege votre API contre les abus en limitant le nombre de requetes per IP address et per route dans une fenetre de temps donnee.

How it works

The RateLimitMiddleware uses a key composed of the client's IP address and the requested route. Each IP+route combination has its own counter.

  Client 192.168.1.10           Client 192.168.1.20
        │                              │
        v                              v
  ┌──────────────┐              ┌──────────────┐
  │ GET /api/users│              │ GET /api/users│
  │  Counter: 28/30│            │  Counter: 5/30│
  └──────┬───────┘              └──────┬───────┘
         │                              │
         v                              v
  ┌────────────────────────────────────────────┐
  │           RateLimitMiddleware               │
  │  Key = sha256(IP + Route)                  │
  │  Store: Redis (prod) / InMemory (dev)     │
  └────────────────────────────────────────────┘
Stores: Redis vs InMemory
StoreContextAdvantage
RedisStoreProduction, KubernetesShared between all pods/workers. Counters are synchronized via Redis.
InMemoryStoreDevelopmentZero external dependency. Counters live in the worker memory.
Watch out in dev. With InMemoryStore, each worker has its own counters. If you have 4 workers, a client can theoretically make 4x the limit.
Configuration per route

Add the middleware with its parameters directly on a route:

use Framework\Middleware\RateLimitMiddleware;

// 30 requetes max per window de 60 secondes
$router->post(
    '/api/login',
    [TokenController::class, 'generate'],
    [[RateLimitMiddleware::class, ['limit' => 30, 'window' => 60]]]
);

// Stricter limit for sensitive endpoints
$router->post(
    '/api/password/reset',
    [PasswordController::class, 'reset'],
    [[RateLimitMiddleware::class, ['limit' => 5, 'window' => 300]]]
);
Attribut #[RateLimit]

You can also use the PHP attribute directly on the controller method:

use Framework\Attributes\RateLimit;

class TokenController
{
    #[RateLimit(60, 'minute')]
    public function generate(): array
    {
        // 60 requetes per minute pour cet endpoint
        return ['token' => $token];
    }

    #[RateLimit(10, 'hour')]
    public function refresh(): array
    {
        // 10 requetes per hour
        return ['token' => $newToken];
    }
}
Response headers

Each response contains headers indicating the rate limiting state:

HeaderDescriptionExample
X-RateLimit-LimitMaximum number of allowed requests30
X-RateLimit-RemainingRemaining requests in the window27
X-RateLimit-ResetUnix timestamp of reset1711036800
429 Too Many Requests Response

When the limit is reached, the middleware returns a 429 response with a JSON body:

// HTTP/1.1 429 Too Many Requests
// X-RateLimit-Limit: 30
// X-RateLimit-Remaining: 0
// X-RateLimit-Reset: 1711036800
// Retry-After: 42

{
    "error": "Too Many Requests",
    "message": "Rate limit exceeded. Retry after 42 seconds.",
    "retry_after": 42
}
Do not ignore the Retry-After header. Well-implemented clients should respect this header to avoid being blocked longer. Continuing to send requests after a 429 can result in a temporary ban at the reverse proxy level.

3 Security Headers Beginner

Le SecurityHeadersMiddleware ajoute automatically des headers de securite a chaque reponse HTTP. Il est enregistre comme middleware global, donc toutes les routes en beneficient sans configuration supplementaire.

Added headers
HeaderValueProtection
X-Content-Type-OptionsnosniffPrevents the browser from guessing the MIME type
X-Frame-OptionsDENYBlocks inclusion in an iframe (anti-clickjacking)
X-XSS-Protection1; mode=blockActivates the browser XSS filter
Strict-Transport-Securitymax-age=31536000; includeSubDomainsForces HTTPS for 1 year (HSTS)
Content-Security-Policydefault-src 'self'Limits allowed content sources (CSP)
Referrer-Policystrict-origin-when-cross-originControls information sent in the Referer header
Permissions-Policygeolocation=(), camera=(), microphone=()Disables sensitive browser APIs
Global registration

The middleware is declared in the global middleware configuration. It runs for every request:

// config/middleware.php
return [
    ProfilerMiddleware::class,
    SecurityHeadersMiddleware::class,   // <-- global, all routes
    RequestLoggingMiddleware::class,
    RateLimitMiddleware::class,
];
Pas de configuration necessaire. Le middleware utilise des valeurs by default securisees. Pour une API JSON, ces headers suffisent dans la grande majorite des cas. Si vous servez aussi du HTML, ajustez la CSP selon vos besoins.
Internal implementation 
namespace Framework\Middleware;

class SecurityHeadersMiddleware
{
    public function handle(callable $next): mixed
    {
        // Execute the request normally
        $response = $next();

        // Add security headers
        header('X-Content-Type-Options: nosniff');
        header('X-Frame-Options: DENY');
        header('X-XSS-Protection: 1; mode=block');
        header('Strict-Transport-Security: max-age=31536000; includeSubDomains');
        header('Content-Security-Policy: default-src \'self\'');
        header('Referrer-Policy: strict-origin-when-cross-origin');
        header('Permissions-Policy: geolocation=(), camera=(), microphone=()');

        return $response;
    }
}

4 Request Logging Beginner

The RequestLoggingMiddleware traces each incoming request with a unique identifier, measures the duration and memory consumed, then logs the result with the appropriate severity level.

Request ID

Each request receives a unique identifier (UUID v4) that is propagated in the logs and returned to the client via the X-Request-Id header. This allows tracing a request end-to-end.

# Reponse HTTP
HTTP/1.1 200 OK
X-Request-Id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
Content-Type: application/json
Distributed debugging. In case of a problem, share the X-Request-Id with your backend team. They can instantly find all logs associated with that request.
Log format

Each request is logged with the following format:

METHOD URI STATUS DURATION

# Examples:
[2026-03-21 14:30:12] INFO  GET /api/users 200 12.4ms 2.1MB req_a1b2c3d4
[2026-03-21 14:30:13] WARN  POST /api/login 401 3.2ms 1.0MB req_e5f67890
[2026-03-21 14:30:15] ERROR GET /api/reports 500 45.1ms 8.3MB req_abcdef12
Log levels

The middleware chooses the log level based on the HTTP status code:

Code HTTPLevelMeaning
2xx / 3xxinfoRequest processed successfully or redirect
4xxwarningClient error (validation, auth, not found)
5xxerrorServer error (unhandled exception, bug)
Collected metrics

The middleware measures two key metrics for each request:

  • Duration: time elapsed between middleware entry and exit (in milliseconds)
  • Memory: peak memory used during request processing (in megabytes)
namespace Framework\Middleware;

class RequestLoggingMiddleware
{
    public function handle(callable $next): mixed
    {
        $requestId = bin2hex(random_bytes(16));
        $start     = hrtime(true);

        // Propagate the request ID
        header("X-Request-Id: $requestId");

        // Execute the rest of the pipeline
        $response = $next();

        // Calculate the metrics
        $duration = (hrtime(true) - $start) / 1e6;
        $memory   = memory_get_peak_usage(true) / 1024 / 1024;
        $status   = http_response_code();

        // Choose the log level
        $level = match(true) {
            $status >= 500 => 'error',
            $status >= 400 => 'warning',
            default        => 'info',
        };

        // Log the request
        Logger::log($level, sprintf(
            '%s %s %d %.1fms %.1fMB %s',
            $_SERVER['REQUEST_METHOD'],
            $_SERVER['REQUEST_URI'],
            $status,
            $duration,
            $memory,
            $requestId
        ));

        return $response;
    }
}
