1 Audit Trail (SOC 2) Intermediate

L'audit trail enregistre automatically toutes les modifications (creation, mise a jour, suppression) des models annotes. Conforme aux exigences SOC 2 pour la tracabilite des donnees.

#[Auditable] Attribute 
namespace App\Models;

use Fennec\Core\Model;
use Fennec\Core\Audit\HasAuditTrail;
use Fennec\Core\Audit\Auditable;

#[Auditable(except: ['password', 'remember_token'])]
class User extends Model
{
    use HasAuditTrail;
}

// Chaque modification est automatically tracee :
$user = User::find(1);
$user->name = 'New name';
$user->save();
// → INSERT INTO audit_logs (model, model_id, action, changes, user_id, ip, ...)
audit_logs table structure
ColumnTypeDescription
idSERIALUnique identifier
model_typeVARCHARModel name (ex: App\Models\User)
model_idINTEGERModified entity ID
actionVARCHARcreated / updated / deleted
changesJSONBOld and new values
user_idINTEGERAuthor of the modification
ip_addressVARCHARClient IP address
created_atTIMESTAMPAction date
 
# Generate the complete audit module
./forge make:audit

# Purge old logs (ISO 27001 retention)
./forge audit:purge --days=365 --dry-run

2 NF525 — Fiscal Compliance Advanced

The NF525 module guarantees invoice immutability and fiscal traceability via a SHA-256 hash chain. Each invoice is linked to the previous one, making any modification detectable.

#[Nf525] Attribute and hash chain 
namespace App\Models;

use Fennec\Core\Model;
use Fennec\Core\Nf525\HasNf525;
use Fennec\Core\Nf525\Nf525;

#[Nf525(prefix: 'FA')]
class Invoice extends Model
{
    use HasNf525;
}

// Hash chain:
// hash(FA-0001) = SHA256(data + "0000...0000")
// hash(FA-0002) = SHA256(data + hash(FA-0001))
// hash(FA-0003) = SHA256(data + hash(FA-0002))
// → Toute modification casse la chaine
  FA-0001              FA-0002              FA-0003
  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ data     │         │ data     │         │ data     │
  │ prev: 00 │────────►│ prev: a1 │────────►│ prev: b2 │
  │ hash: a1 │         │ hash: b2 │         │ hash: c3 │
  └──────────┘         └──────────┘         └──────────┘

  Verification: recompute each hash and compare
  ./forge nf525:verify --table=invoices
Closings and FEC export 
# Periodic closings
./forge nf525:close --daily=2026-03-22
./forge nf525:close --monthly=2026-03
./forge nf525:close --annual=2026

# FEC export (Accounting Entries File)
./forge nf525:export --year=2026 --output=fec_2026.txt

# Verify chain integrity
./forge nf525:verify --table=invoices
Strict immutability. Once created, an NF525 invoice can never be modified or deleted. To correct an error, create a credit note via $invoice->createCredit().

3 Encryption AES-256-GCM Intermediate

The #[Encrypted] attribute transparently encrypts sensitive database fields. Encryption uses AES-256-GCM with a key defined in environment variables.

Utilisation 
namespace App\Models;

use Fennec\Core\Model;
use Fennec\Core\Encryption\HasEncryptedFields;
use Fennec\Core\Encryption\Encrypted;

class Patient extends Model
{
    use HasEncryptedFields;

    #[Encrypted]
    public string $ssn;          // Social security number

    #[Encrypted]
    public string $medical_notes; // Medical notes

    public string $name;          // Not encrypted
}

// Transparent usage:
$patient = new Patient();
$patient->ssn = '1 85 12 75 108 042 35';
$patient->save();
// In DB: ssn = "encrypted:base64data..."

$patient = Patient::find(1);
echo $patient->ssn;
// → "1 85 12 75 108 042 35" (dechiffre automatically)
# .env — Encryption key (32 bytes, base64)
ENCRYPTION_KEY=base64:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx=
Encryption key. Losing the ENCRYPTION_KEY makes all encrypted data irrecoverable. Save it in a secret manager (Vault, K8s Secrets) and never in the source code.

5 Multi-Tenancy Advanced

Le TenantManager supporte l'isolation par base de donnees (database-per-tenant). Chaque tenant est resolu automatically par son domaine et connecte a sa propre base.

Configuration 
# Generate the configuration file
./forge make:tenant

# app/config/tenants.php :
return [
    'resolver' => 'domain',  // Resolution by domain
    'tenants' => [
        'acme' => [
            'domain'   => 'acme.example.com',
            'database' => 'db_acme',
        ],
        'globex' => [
            'domain'   => 'globex.example.com',
            'database' => 'db_globex',
        ],
    ],
];
Complete isolation. Each tenant has its own database. Migrations run for each tenant independently. No data can leak between tenants.

6 Security — Summary Beginner

Fennec integrates multiple security layers that can be enabled by configuration or attribute:

Security components
ComponentDescriptionConfiguration
PasswordPolicy Password validation (length, complexity) PASSWORD_MIN_LENGTH (defaut: 8)
AccountLockout Lockout after N failed login attempts LOCKOUT_MAX_ATTEMPTS=5, LOCKOUT_DURATION=900
SecurityLogger Security event logs (auth, tokens) Writes to var/logs/security.log + stderr
IpAllowlistMiddleware Access restriction by IP/CIDR IP_ALLOWLIST (comma-separated)
LogMaskingProcessor Automatic masking of sensitive data in logs Actif by default (password, token, secret, etc.) 
use Fennec\Core\Security\PasswordPolicy;
use Fennec\Core\Security\AccountLockout;
use Fennec\Core\Security\SecurityLogger;

// Validate a password
$errors = PasswordPolicy::validate($password);
if (!empty($errors)) {
    return ['errors' => $errors];
}

// Check the lockout
if (AccountLockout::isLocked($email)) {
    SecurityLogger::alert('auth.locked', ['email' => $email]);
    return ['error' => 'Account locked'];
}

// Record a failure
AccountLockout::recordFailure($email);

// Reset after successful login
AccountLockout::reset($email);
Defense en profondeur. Chaque couche de securite est independante. Meme si une couche est contournee, les autres protegent l'application. Activez-les toutes in production.
